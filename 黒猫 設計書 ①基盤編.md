# 黒猫 設計書 ① 基盤編 v1.0

税務調査業務を支える常駐トレイアプリ「黒猫」の設計書、全4編のうち第1編。
本編は (A) プロジェクト全体像とインデックス、(B) 常駐シェル（親）の設計、
(C) コマンドパレット（司令塔）の設計、(D) 全機能を貫く共通規約を収める。
機能編（②）・連携拡張編（③）は本編の共通規約を参照する。

- プロジェクト: 黒猫 ―― 単一トレイ常駐プロセスに複数機能を統合
- 対象OS: Windows 10 / 11 ／ 言語: Python 3.x（フルインストーラ版）
- 不変原則: 外部通信ゼロ / 機微非保持 / ローカル完結 / 疎結合 / PowerShell非依存
- 本編が正本とする領域: モード調停・フック調停・ホットキー・設定・潔白性・
  共通Win32/DPI・コマンド供給IF（機能編・連携拡張編はこれを参照）

-----

> ## ★実物向けの読み替え（実装担当は最初に読むこと）
> 
> **黒猫は既に存在し、トレイ常駐のランチャーとして稼働している**（眼鏡＋鈴の

## A0. このプロジェクトは何か

職場（税務署）の厳格な制約（外部通信禁止・ExecutionPolicy=Restricted・
CSV出力不可等）のもとで、PC作業を支える常駐ツール群を**単一のトレイ常駐
プロセス**に統合したもの。黒猫という1匹の常駐本体が、複数の道具（機能）を
提げている構造。

設計の狙い:

1. ツール乱立（個別プロセス・トレイ・ホットキーの散在）を1つに束ねる。
1. 各機能を疎結合に保ち、機能追加が局所変更で済む。
1. 潔白性（機微を残さない・外部に出さない）をシェル層で全機能に強制する。

# 第B部　常駐シェル（親）

## B1. 目的とスコープ

複数の常駐ツールを個別プロセスで散在させず、1つのトレイ常駐プロセスに統合
する。トレイアイコン乱立・ホットキー衝突・設定分散・多重起動・モード競合を
一元解決し、機能追加を容易にする。

含む: 単一プロセス常駐 / モジュール登録・ライフサイクル / モード調停 /
ホットキー一元管理 / 共通インジケータ / 共通設定 / 単一インスタンス /
DPI・モニタ・Win32共通化 / フック調停 / 死活監視 / 潔白性強制 / 安全終了。

含まない: 各機能の具体ロジック（②機能編）/ 自動更新・配布 / 作業データの
永続化（設定のみ永続）。

優先順位: ①安全と潔白性 ②単一性（1プロセス・1トレイ・1対話モード）
③堅牢性（1機能の不調が波及しない）④拡張性（局所変更で機能追加）。

## B2. 設計思想

- **黒猫が道具を提げる一体型**: 本体が常駐し、機能はその上に載る。機能同士は
  直接依存せず、シェルが提供するサービス（ホットキー・モード・設定・フック・
  Win32）越しにのみ協調する。1機能の追加・故障が他へ波及しない。
- **対話モードは常に1つ**: 記録・配置・転記等は「クリックやキーを横取りする
  対話モード」を持つ。同時に走ると二重解釈・記録混入が起きる。シェルが同時
  アクティブな対話モードを1つに限定する（§B7）。各機能が個別に書く「排他」の
  正体はこの一機能。
- **フックは共有資源**: グローバルフックをシェルが一元保持し、アクティブな
  モジュールへ配分する（§B13）。
- **潔白性をシェルが強制**: 外部通信ゼロ・機微非保持等を機能任せにせず
  シェル層で担保・検査する（§B15.4）。どの機能を足しても掟は破れない。

## B3. 全体アーキテクチャ

```
┌───────────────────────────────────────────────────────────┐
│ 黒猫シェル（単一プロセス / pythonw.exe）                     │

│  [共通インジケータ] 現在モード表示（§B9）                   │
│  ┌── コア・サービス層（機能を知らない・疎結合） ─────────┐ │
│  │ モード調停(§B7)/ホットキー(§B8)/設定(§B10)             │ │
│  │ 単一インスタンス(§B11)/Win32・DPI・モニタ(§B12)        │ │
│  │ フック調停(§B13)/死活監視(§B14)/ログ(§B15)            │ │
│  └────────────────────────────────────────────────────────┘ │
│  ┌── 機能モジュール層（プラグイン的） ──────────────────┐ │
```

コア層は機能を知らない。機能はコアの公開IF（§B5.2）を使う。機能の増減は
モジュール層の差し替えで済み、コア層を変更しない。

## B4. スレッド／プロセスモデル

|スレッド      |責務                                  |
|----------|------------------------------------|
|メイン（UI）   |tkinter mainloop・トレイ・インジケータ・各機能ダイアログ|
|フック       |pynput Listener。イベントを調停経由で配分（§B13）  |
|ワーカー（機能ごと）|重い処理（UIA・スクショ整形・生成）。機能が必要時に持つ       |
|監視        |死活監視・ホットキー健全性・メモリ監視の軽量タイマー（§B14）    |

- スレッド間はキューのみ。共有可変状態を避ける。
- 例外: 一部機能はフックスレッドでの同期処理を要する（記録のスクショ同期
  撮影等）。シェルが「短時間で返す同期実行枠」を提供して許可（§B13.3）。

## B5. モジュール・モデル（拡張の要）

### B5.1 ライフサイクル

|段階        |内容                                                    |
|----------|------------------------------------------------------|
|register  |起動時に自己登録（名前・メニュー項目・希望ホットキー・対話モード有無・コマンド供給）（§B5.2, §C6）|
|activate  |対話モード開始（調停器の許可が前提・§B7）                                |
|on_event  |アクティブ中、配分されたマウス/キーを処理（§B13）                           |
|deactivate|対話モード終了・横取り解除                                         |
|shutdown  |プロセス終了時の後始末（フック解除・一時状態破棄・§B16）                        |

### B5.2 シェルが公開するサービス（機能から呼ぶIF）

- モード取得・要求・解放（§B7）
- ホットキー登録・解除（§B8）
- 設定の取得・保存（§B10）
- インジケータへの状態通知（§B9）
- Win32／DPI／モニタ情報・キー送出・クリップボードの共通ユーティリティ（§B12）
- ログ出力（§B15）
- **コマンド供給**（コマンドパレットへコマンドを申告・§C6。連携拡張編③で
  全機能が利用）
- **連携IF**（テキスト受け渡し・ウィンドウ委譲・起動委譲・連携調停。③で定義し
  本IFに集約）

### B5.3 機能追加の局所性

新機能は「モジュール規約（§B5.1）を実装し register するだけ」で載る。コア層・
既存機能には触れない。トレイメニューとホットキーは register 情報から自動組込
（§B6.2, §B8.2）。

## B6. トレイアイコンとメニュー

- 単一トレイアイコン（黒猫の意匠）。現在モードで図柄/色を変える（待機=座り猫、
  対話モード中=目が光る等。インジケータ§B9と整合）。左=クイック、右=フル。
- メニューは register 情報から自動生成。各機能ブロックが差し込まれ、機能追加で
  自動的に増える。対話モード中は競合する他機能項目をグレーアウト（§B7.3）。

```
黒猫
├─［状態表示］現在: 待機 / 記録中 / 配置中 / 転記中 …
├─ ManualRecorder ▸ / ウィンドウ配置 ▸ / 項目転記 ▸ / …（各機能が差込）
├─ 設定… / バージョン情報 / 終了
```

## B7. モード調停（同時に1対話モードだけ）

### B7.1 定義

クリック/キーを横取りして独自解釈する状態。記録(RECORDING)・配置・転記の
収集/流し込み等が該当。各機能は対話モードに入る前にシェルへモード要求する。

### B7.2 調停規則（既定・設定可）

|現モード|新要求  |既定の挙動                        |
|----|-----|-----------------------------|
|なし  |任意   |許可してアクティブ化                   |
|記録中 |配置/転記|記録を一時停止(PAUSED)→新モード。終了時に再開確認|
|配置中 |記録/転記|配置を終了してから新モード（ピンは維持）         |
|転記中 |記録/配置|転記フェーズ終了を促す（収集中なら確認）→新モード    |

「記録中だけは一時停止して退避」を既定にするのは、記録の連続性が他より価値が
高く再開可能だから。配置・転記は状態が軽く損失が小さい。

> 機能側設定 `recording_conflict`（②の配置・転記等が持つ）はこの調停の機能別
> 上書き。値 `pause`＝上表どおり記録を一時停止して新モードへ（既定）、`deny`＝
> 記録中はその機能の起動を拒否し「記録中は使用できません」を表示（記録を一切
> 乱したくない場合）。本§B7が正本で、各機能の同設定はここに従う。

### B7.3 競合の可視化

対話モード中は衝突する他機能のメニュー・ホットキーを無効化、インジケータに
現モードを明示。無効化操作には「○○中は使用できません」を短く通知。

### B7.4 記録への混入防止（横断要件）

他機能が送出するキー（転記のCtrl+C/V/Tab、配置のフォーカス変化等）は、
ManualRecorderの記録対象から除外する。除外はシェルが「ツール起因」と
マークして配分段階で記録モジュールに渡さないことで実現（§B13.4）。各機能が
個別に書く除外要件の実装責任をここで一元化する。

## B8. ホットキー一元管理

- 全ホットキーをシェルが一括登録。機能は register 時に「希望キー＋用途」を申告。
- **衝突検出**: 機能間でキー重複を起動時に検出して通知、後勝ち/設定優先で解決。
  Windows標準（Win+矢印等）や対象アプリ主要ショートカットと当たりにくい既定。
- 正本は第A部§A4。機能追加時はそこへ追記し衝突検出で安全担保。

## B9. 共通インジケータ

現在モードを常時可視化する小ウィンドウ。全機能で共有し乱立させない。

|モード   |色    |文言例             |
|------|-----|----------------|
|待機    |灰    |黒猫：待機中          |
|記録中   |赤    |●記録中　F8:注釈 F9:終了|
|記録一時停止|黄    |‖一時停止           |
|配置中   |青    |配置モード：対象モニタN    |
|転記収集  |緑    |収集中：N件          |
|転記流し込み|緑(点滅)|流し込み中…          |

機能は§B5.2のIFで「状態テキスト・色」を通知するだけ。描画はシェル。F12で
表示/非表示・ドラッグ移動可・位置は設定に永続化。

## B10. 共通設定基盤

- 設定は単一ローカルファイル（例: settings.json）。外部に出さない。
- 名前空間分離: `shell` / 各機能名（`recorder`/`placement`/`relay`…）。シェルは
  器として供給し中身を解釈しない。キー名は各機能設計の設定表に一致。
- トレイ「設定…」から機能別タブの設定UI。不正値は既定へフォールバックし起動を
  止めない。
- 永続化するのは**設定のみ**。記録データ・配置ピン・転記バッファ等は永続化
  しない（各機能の方針を尊重）。
- **永続ファイルの配置規約**（②各機能の `persist_file`/`save_dir` 等の「ローカル
  既定」の具体化）: 既定は `%APPDATA%\Kuroneko\` 配下とし、機能ごとに
  サブファイル（例 `notes.json`/`search_index.db`/`captures\`）。設定ファイルと
  同じ親フォルダに置き、外部に出さない（①§D1）。各機能設計の「ローカル既定」は
  本規約を指す。

## B11. 単一インスタンス保証

多重起動を防ぐ（フック多重・トレイ重複・ホットキー二重登録を回避）。既存
インスタンスがある状態で再起動されたら、新規は既存へ「前面化/メニュー表示」を
依頼して自身は終了。名前付きミューテックス等のOS機構（外部通信を伴わないもの）。

## B12. 共通Win32 / DPI / モニタ基盤

- **DPI**: Per-Monitor DPI Aware v2 をシェルが一括設定。各機能は物理座標で一貫
  処理（配置の矩形計算・スクショ座標・パネル位置）。
- **共通ユーティリティ**（機能から呼ぶ・重複実装させない）:
  - 前面ウィンドウ情報（タイトル・exe・クラス名）取得
  - モニタ列挙・作業領域・カーソル所在モニタ判定
  - キー送出（SendInput ラッパ）／クリップボード退避・設定・復元
  - ウィンドウ操作（移動・リサイズ・topmost・前面化）ラッパ
  - スクリーンキャプチャ（記録・領域キャプチャが共有）
  - 共通整形エンジン（履歴・転記が共有・③第1波で追加）

## B13. フック調停（入力フックの共有）

- **一元保持**: マウス/キーボードのグローバルフックをシェルが1組だけ保持。機能
  ごとに張らない（多重フック・解除漏れ防止）。
- **ディスパッチ**: 現在アクティブな対話モジュールにのみイベントを配分。待機中の
  素の操作は誰にも配分しない。
- **同期実行枠**: フックスレッドでの短時間同期処理を要する機能（記録のスクショ
  同期撮影等）に、応答性を壊さない範囲（数十ms）で枠を提供（§B13.3）。
- **ツール起因の標識**（§B13.4）: 機能が送出したキー（転記のCtrl+C/V/Tab等）に
  「自ツール起因」標識を付け、記録モジュールへ配分しない。自動操作が手順に
  混入しない（§B7.4の実装）。

## B14. 死活監視・自己防衛

- 監視項目: フック健全性（対話中なのに一定時間イベントが来ない→遮断の可能性）/
  ホットキー登録の生存 / メモリ使用（記録の長時間化等）→警告 / アクティブ
  モジュールのワーカー応答性。
- 異常時: 1機能の例外でプロセス全体を落とさない。当該機能を安全に deactivate し、
  シェルと他機能は生存継続。フックが死んだら再登録を試み、不可ならトレイ通知。

## B15. エラーハンドリング・ロギング・潔白性の強制

- **障害局所化**: モジュール呼び出しを例外境界で包み、1機能の失敗を波及させ
  ない。コア層の致命的異常のみ安全終了（§B16）へ。
- **ロギング**: ローカルファイルへ最小限（時刻・モード遷移・例外）。機微情報
  （文字入力内容・クリップボード中身・スクショ・転記値・ファイルパス内の機微）は
  書かない。ログ自体も外部送信しない。
- **設定不正への耐性**: 破損・不正値は既定へフォールバックし起動継続。
- **潔白性の集中強制（§B15.4）**: シェルが全機能に対し次を保証・検査する。
  - 外部通信を行うコードを含まない（依存も通信機能を使わない）。
  - 文字入力内容・パスワードを保持しない（記録機能の原則をシェルも監督）。
  - クリップボードは借りたら返す（退避・復元をシェルが保証）。
  - 生成物・ログに機微情報を残さない。

> どの機能を足してもこの掟は破れない。新機能のレビュー観点も本節を基準とする。

## B16. 起動・終了シーケンス

**起動**: ①単一インスタンス確認（多重なら既存を前面化して終了）②DPI Aware設定
③設定読込（破損は既定）④コア初期化（調停・ホットキー・フック・監視）⑤各
モジュール register→トレイ・ホットキー組込 ⑥トレイ常駐開始・待機状態。

**終了**: ①アクティブ対話モジュールを deactivate ②全モジュール shutdown（一時
状態破棄・ピン解除等は各機能規約）③フック解除・ホットキー解除・トレイ除去
④設定保存（インジケータ位置等）⑤単一インスタンス標識解放。記録中など未保存の
作業があれば確認（破棄/保存）を挟む。

# 第C部　コマンドパレット（司令塔）

═══════════════════════════════════════════════════════════════════

## C1. 目的とスコープ

機能が増えホットキーが乱立して覚えきれなくなる問題を解消する。**単一の入口**
（Ctrl+Alt+P）から機能名・アクション名をあいまい検索して実行でき、ホットキーを
暗記しなくても全機能に到達できる。

含む: 単一ホットキー起動 / 全モジュールのコマンド横断集約（register連携）/
あいまい検索（ローマ字・かな・部分一致・頭文字）/ 最近・よく使う順 / 引数つき
コマンドの2段入力 / 直前ウィンドウへの復帰。

含まない: 自然言語の自由指示（語彙ベース）/ マクロ（③連携拡張編）/ 外部アプリ
コマンド取り込み（黒猫内のみ）。

優先順位: ①速さ ②網羅（全機能・主要アクションへ到達）③潔白性（クエリ・履歴に
機微を残さない）④学習による快適さ。

## C2. 設計思想

- **司令塔＝register の上に立つメタ機能**: シェルの登録機構（§B5）の上に乗り、
  登録情報を集約して入口を1つに統一する。機能が増えるたびにパレットへ自動で
  並ぶため追加コストが逓減。新機能側は「コマンドを申告するだけ」。
- **ホットキーは近道・パレットは正面玄関**: 個別ホットキー（Ctrl+Alt+W等）は熟練
  者の近道として残し、パレットは「全部ここから行ける」正面玄関。暗記負担を
  パレットが肩代わり（§C2.2）。
- **機能だけでなくアクションまで**: 「配置を開く」だけでなく「全付箋を退避」
  「履歴を全消去」「スロット1を呼び出す」等の主要アクションもコマンド化して
  操作系を統一（§C6.2）。
- **潔白性**: 検索クエリに機微が打ち込まれうる。クエリ・実行内容はログに残さず、
  永続化するのはコマンドの使用頻度だけ（§C9.2）。

## C3. 起動とホットキー

|操作        |機能                |
|----------|------------------|
|Ctrl+Alt+P|パレット表示/非表示トグル     |
|入力        |あいまい検索（§C8）       |
|↑↓ / Tab  |候補移動              |
|Enter     |実行（引数要なら2段目へ・§C10）|
|Esc       |閉じる（2段目なら1段目へ戻る）  |

パレットは最前面・即フォーカス・直前ウィンドウを記憶（§C11）。

## C4. システム構成

```
コマンドパレット
├─ コマンドレジストリ: 全モジュールから集約したコマンド表（§C6）
├─ 検索エンジン: あいまい照合＋スコアリング（§C8,§C9）
├─ パレットUI: 入力欄＋候補リスト＋2段入力（§C7,§C10）
└─ 実行系: コマンド起動・引数受け渡し・フォーカス復帰（§C11）
```

## C5. パレットUI

- 画面中央上寄りの入力欄＋候補リスト（VS Code風）。1行＝コマンド: アイコン
  （group色）・title・group名・割当ホットキー（あれば）。
- **ホットキーの可視化**: 各行にそのコマンドのホットキーを併記。使ううちに自然に
  覚え、近道へ移行できる導線。
- 無入力時は「最近使った」「よく使う」を上に、続いてgroupごと代表コマンド。
  入力が進むとフラットなスコア順へ。

## C6. コマンド供給（register連携・本機能の核）

### C6.1 供給API

各モジュールが register 時（§B5.1）にシェル経由でコマンドを申告。

|項目          |説明                                  |
|------------|------------------------------------|
|id          |一意ID（例: placement.start）            |
|title       |表示名（例: ウィンドウ配置を開始）                  |
|keywords    |検索補助語（かな/ローマ字/別名。例: はいち, haichi, 整列）|
|group       |所属機能                                |
|needs_args  |引数が必要か（§C10）                        |
|arg_provider|動的引数の供給関数参照（§C6.3, §C10）            |
|enabled_when|実行可否の条件（現モード等・§C6.4）                |
|run         |実行ハンドラ参照                            |

### C6.2 機能アクションまでコマンド化（例）

|group         |コマンド例                           |
|--------------|--------------------------------|
|ManualRecorder|記録を開始 / 終了して保存 / 直前手順に注釈 / メモ挿入 |
|ウィンドウ配置       |配置モード開始 / 全固定解除 / 固定中ウィンドウ一覧    |
|ウィンドウ呼び出し     |スロット{n}を呼び出す（動的）/ 呼び出しパレット      |
|項目転記          |収集を開始 / 流し込み実行 / 収集パネル表示        |
|クリップボード履歴     |履歴パネル / 履歴を全消去 / 監視を停止・再開       |
|付箋ピン          |新規付箋 / 選択から付箋 / 全付箋を退避・復帰 / 付箋一覧|
|領域キャプチャ       |領域選択 / 全退避 / 全消去                |
|ファイル検索        |ファイル検索 / 再インデックス / インデックス消去     |
|作業状態          |作業状態を保存 / 復元（一覧・動的）             |
|シェル           |設定を開く / バージョン情報 / 黒猫を終了         |

### C6.3 動的コマンド

実行時状況で内容が変わるものは arg_provider が候補を返す（スロット一覧・付箋
一覧・履歴上位・作業状態スロット一覧等）。2段目（§C10）で選ばせる。

### C6.4 実行可否（enabled_when）

現在のモード・状態で実行できないコマンドは、候補に出すが淡色＋実行時に理由
表示、または非表示（設定）。例: 「流し込み実行」はバッファが空なら淡色、「記録
を終了」は記録中のみ。モード調停（§B7）と整合。

## C7. 引数つきコマンドの2段入力

1. 1段目でコマンド選択（needs_args=true）。
1. arg_provider が候補/入力形式を返す→2段目表示。
1. 2段目で選択/入力→実行。

- 候補リスト型（スロット一覧・付箋一覧・履歴上位）／自由入力型（付箋本文等）。
  自由入力の内容は実行に使うだけで**保存しない**（§C9.2）。
- 2段目で Esc→1段目へ戻る。もう一度 Esc で閉じる。

## C8. あいまい検索

- 照合対象は title・keywords・group。keywords に かな/ローマ字/別名 を持たせ
  表記ゆれを吸収（§C6.1）。
- 部分一致・連続部分列（fuzzy）・頭文字に対応。「はいち/haichi/整列」→配置、
  「ふせん/fusen/メモ」→付箋、「りれき/rireki/clip」→履歴。
- 日本語は keywords で読みを吸収（自動読み推定はせず登録側が読みを入れる方針で
  確実性を担保）。
- スコア = 一致の良さ（先頭>連続>飛び飛び）＋使用頻度（§C9）＋実行可否。

## C9. 学習（最近／よく使う）

- スコア要素: 使用回数・最近性・直近文脈（任意・弱め。例「呼び出し」直後は
  「配置」系を微加点）。
- **永続化の範囲**: コマンドidごとの使用回数・最終使用時刻のみ。検索クエリの
  文字列・引数の機微値は残さない（§C9.2）。

## C10. 引数つきコマンドの2段入力（§C7参照）

本機能の2段入力の仕様は §C7 に記す。本文中の「§C10」表記は §C7 と読み替える
（2段入力の詳細＝候補リスト型/自由入力型・Escで戻る・自由入力は保存しない）。

## C11. 実行とフォーカス復帰・潔白性

- 起動時に直前フォアグラウンドを記憶。フォーカス不要なコマンドは実行後に直前
  ウィンドウへ戻す（パレットが作業ウィンドウのフォーカスを奪いっぱなしにしない）。
- 対話モードを開始するコマンド（記録開始・配置開始等）は当該機能へ制御を渡し、
  モード調停（§B7）に従う。送出キーは記録除外（§B13.4）。
- 検索クエリ・2段目自由入力は保存もログもしない。永続はコマンド使用頻度のみ。

## C12. 親（シェル）への要請（反映済み前提）

本機能の実現に、§B5.2 公開サービスへ「コマンド供給」を1項目追加する。動的
コマンド（arg_provider）と実行可否（enabled_when）を提供できるIFも含む。影響は
加算的で既存機能の動作を変えない。各既存モジュールは「コマンド申告」を足すだけで
パレット対応になる（後方互換）。

## C13. コマンドパレット 設定・エッジケース・チェックリスト

設定: hotkey_palette(Ctrl+Alt+P) / show_disabled(dim|hide) / recent_count(5) /
frequent_count(5) / learning_enabled(true) / context_boost(true) /
restore_focus(true) / max_results(12)。

エッジケース: runが例外→失敗通知・他は継続 / enabled_when偽→理由表示 / 動的
引数取得失敗→1段目へ / 0件→「一致なし」＋近い候補提案 / 重複id→後勝ち＋警告 /
表示中の再ホットキー→閉じる。

チェックリスト:

- [ ] register経由のコマンド供給API（§C6.1）
- [ ] 機能起動＋機能内アクションのコマンド化（§C6.2）
- [ ] 動的コマンド（arg_provider）・実行可否（enabled_when）（§C6.3,§C6.4）
- [ ] あいまい検索（部分/fuzzy/頭文字・かな/ローマ字をkeywordsで吸収）（§C8）
- [ ] スコアリング（一致良さ＋頻度＋可否）（§C8）
- [ ] 最近/よく使う学習（頻度のみ永続）（§C9）
- [ ] パレットUI・即フォーカス・ホットキー併記（§C5）
- [ ] 2段入力（候補型/自由入力型・Escで戻る）（§C7）
- [ ] 実行後フォーカス復帰（§C11）
- [ ] クエリ/自由入力を保存・ログしない（§C9,§C11）
- [ ] モード調停・記録除外の遵守（§C11）

# 第D部　全機能を貫く共通規約（機能編・連携拡張編が参照）

機能編②・連携拡張編③の各機能・各拡張は、以下を**共通の前提**とする。各機能

## D1. 不変原則（全機能必達）

|原則           |内容                     |根拠    |
|-------------|-----------------------|------|
|外部通信ゼロ       |ネットワークアクセスを行うコードを含まない  |§B15.4|

|PowerShell非依存|python.exe直接起動         |§B16  |

|停止しない        |1機能の失敗で全体を落とさない        |§B15.1|

## D2. 共通ライフサイクルと連携の作法

- 各機能は §B5.1 のモジュール規約（register/activate/on_event/deactivate/
  shutdown）に従う。
- 対話モードを持つ機能は §B7 の調停に従い、開始前にモード要求する。
- 機能間連携は §B5.2 の連携IF経由（直接依存しない）。連携が対話モードやキー
  送出を伴う場合もモード調停に従う（詳細は③）。
- 機能が送出する自動キーは「ツール起因」標識で記録除外（§B13.4）。

## D3. 共通の依存・起動・導入（各機能で重複記載しない）

|種別                     |内容                                    |
|-----------------------|--------------------------------------|
|Python                 |フルインストーラ版3.x（tkinter同梱必須・embeddable不可）|
|tkinter                |UI（パネル・ダイアログ・インジケータ） 全機能必須            |
|pynput                 |グローバルフック・ホットキー・キー送出 全機能必須             |
|ctypes(Win32)          |ウィンドウ・モニタ・DPI・クリップボード・SendInput 標準    |
|Pillow                 |スクショ・画像処理（記録・領域キャプチャのみ）               |
|uiautomation(+comtypes)|UIA部品名取得（記録で強く推奨）                     |
|トレイ表示                  |pystray 等（差し替え可能な層）                   |

- 起動は pythonw.exe 直接（PowerShell非依存）。オフライン導入は wheel を一括
  収集して移送（`pip download … -d wheels` → `pip install --no-index --find-links wheels …`）。

## D4. 共通の設定・潔白性の書式

- 設定は §B10 の名前空間に各機能名で格納。キー名は各機能設計の設定表に一致。
- 各機能の「潔白性」記述は §B15.4 を前提に、機能固有の追加点（何を永続し何を
  しないか＝第A部§A5の早見表）だけを各機能設計に記す。

## D5. 共通のインジケータ・トレイ

- 状態表示は §B9 の共通インジケータへ通知するのみ（機能ごとに別表示を作らない）。
- トレイ項目は register で申告し §B6.2 で自動組込。
